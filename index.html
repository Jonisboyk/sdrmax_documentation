funçãoLê a ordem do WR, executa a ferramenta correta e devolve o resultado. Nunca fala com o lead.

loop agêntico · protocolo de fila quando WR envia múltiplas instruções
WR
WR avalia e monta fila de instruções
Quando o lead exige múltiplas ações, o WR emite instruções numeradas e ordenadas por prioridade.
[1] CONSULTAR_RAG | query: produto X características [2] REGISTRAR_CRM | nome: João | interesse: produto X
OP
OP executa [1] e retorna ao WR
Executa apenas a primeira instrução da fila. Nunca encadeia. Devolve resultado ao WR.
{ "ferramenta": "consultar_rag", "parametros": "produto X características" }
WR
WR recebe resultado e emite [2]
Processa o resultado do OP e — se ainda houver instrução na fila — emite a próxima.
OP
OP executa [2] e retorna ao WR
Executa a segunda instrução e devolve resultado via ferramenta: "agent_wr".
✓
WR entrega resposta final ao lead
Fila zerada. WR sintetiza todos os resultados e envia mensagem humanizada. Máx. 10 iterações totais.
02 · limites técnicos
tamanho WR · recomendado
600–800
tokens de conteúdo real. saida_obrigatoria removida do prompt — JSON garantido via response_schema no body da requisição.
⚠ checklist nunca cortar — compensa o thinking OFF
tamanho OP · recomendado
250–400
tokens de conteúdo real. saida_obrigatoria removida — schema do OP também vai no response_schema. OP deve ser enxuto — complexidade vai no WR.
modelo · Gemini 3 Flash
200K
tokens de contexto disponível. Limite de prompt não é o problema — o budget de 600–800 tokens é sobre qualidade de seguimento de instrução, não sobre estourar contexto.
thinking mode
OFF
Desativado por padrão em todos os clientes — menor custo e latência. Ativar apenas quando o projeto específico exigir raciocínio mais profundo.
⚠ thinking OFF = checklist no prompt é obrigatório
histórico enviado ao WR
10 msgs
Últimas 10 mensagens por padrão. Aumentar para 20 em contextos de venda longa sem impacto crítico de custo.
⚠ histórico completo aumenta custo e latência
iterações máximas · loop
10 iter.
Hard limit no n8n. Acima de 10 iterações o loop é interrompido e o handoff humano é acionado automaticamente.
response_schema · JSON
body
Schema separado por agente — WR e OP têm schemas distintos no body da requisição. OP com schema mais estrito evita parametros vazios, o erro silencioso mais comum.
ferramentas por cliente · OP
1–4
Acima de 4 ferramentas o OP começa a confundir qual acionar. Ferramentas definidas no briefing manual — dependem das chaves de integração do cliente.
03 · framework do prompt
<identidade>
variável
✦ extraído do histórico: nome real usado pelo vendedor, empresa, tom predominante
Nome do agente, empresa, papel e tom de voz. Define a persona completa. Tudo vem do briefing + histórico.
<nome>Rafael</nome> <empresa>CompreCar</empresa> <papel>SDR consultivo</papel> <tom>Profissional, humano e objetivo. Linguagem simples e direta.</tom>
<missao>
variável
O que o SDR faz, o que ele jamais faz (limites comerciais) e a metodologia de qualificação usada (BANT, SPIN, etc).
Conversar de forma humanizada, qualificar leads com BANT flexível e acionar o OP quando precisar de dados externos. NUNCA fechar venda, NUNCA negociar condição final.
<objetivos_prioritarios>
fixovariável
Lista ordenada. Os 2 primeiros são fixos em todos os clientes. Os demais variam por objetivo comercial.
1) Responder a dúvida atual do lead com clareza. ← FIXO 2) Conduzir conversa como vendedor real. ← FIXO 3) Acionar OP para consultar catálogo. ← varia 4) Qualificar com BANT no ritmo da conversa. ← varia 5) Acionar OP para registrar lead qualificado. ← varia
<estilo_de_resposta>
do histórico
✦ extraído do histórico: tamanho médio das respostas que converteram, uso de emoji, nível de gíria, padrão de pergunta
Regras de formatação e estilo baseadas nos padrões reais das conversas que funcionaram.
- Respostas curtas por padrão (histórico: média 2,3 linhas nas conversões) - Nível de gíria: baixo - Emoji: raro, nunca repetir o mesmo - 1 pergunta por mensagem
<fluxo_conversa_natural>
do histórico
✦ extraído do histórico Tier 2+3: padrão de abertura, quando qualificar, sequência que levou à conversão
Script não-linear baseado nos padrões reais das conversas que fecharam. Não é um roteiro rígido — é a estrutura que funcionou.
<quando_acionar_operacional>
variáveldo histórico
✦ do histórico: quais perguntas do lead sempre exigiram consulta externa. Quais o vendedor respondia de memória.
Seção mais crítica. Define exatamente quando e como o WR aciona o OP, com o formato da instrução por tipo de ação. Muda completamente por cliente.
<consultar_catalogo> Acione quando: disponibilidade, preço, versão, comparação. Formato: CONSULTAR_CATALOGO | objetivo: [...] | filtros: [...] </consultar_catalogo> <registrar_crm> Acione quando: BANT suficiente + interesse explícito. Formato: REGISTRAR_CRM | nome: [...] | interesse: [...] </registrar_crm>
<objecoes>
do históricoopcional
✦ extraído do histórico: top objeções reais + como o vendedor contornou nas conversas com engajamento genuíno
Seção opcional mas poderosa. Alimentada 100% pelo histórico — não por suposição do cliente no briefing.
<checklist_antes_de_responder> + <restricoes>
fixo
Sempre no final do prompt. Nunca cortar. Com thinking OFF essas seções substituem o raciocínio interno do modelo — são o único mecanismo de auto-verificação antes de responder. O schema JSON vai no response_schema do body da requisição, não aqui.
Budget total estimado do WR: identidade ~80 tokens missao ~60 tokens objetivos ~80 tokens estilo_resposta ~60 tokens ← do histórico fluxo_conversa ~120 tokens ← do histórico quando_acionar_op ~150 tokens ← do histórico + briefing objecoes ~100 tokens ← do histórico (opcional) checklist ~80 tokens ← NUNCA cortar restricoes ~50 tokens ───────────────────────────────── total ~780 tokens
<identidade> + <missao>
variável
Apenas o nome da empresa muda. O papel e a missão do OP são sempre os mesmos: executar, não conversar.
Você é o operacional da [EMPRESA]. Não conversa com o lead. Executa ferramentas e devolve resultado ao [NOME_AGENT_WR].
<ferramentas_disponiveis>
variável
A seção mais variável do OP. Definida inteiramente no briefing manual — depende das chaves de integração que o cliente fornece. Histórico não determina ferramentas, apenas confirma quais dados externos o vendedor precisou consultar.
Ferramentas possíveis por tipo de cliente: - consultar_catalogo → e-commerce, automóveis, imóveis - consultar_rag → base de conhecimento, FAQ, docs - registrar_lead → CRM interno, planilha - registrar_crm → HubSpot, Pipedrive, RD Station - consultar_agenda → agendamento de reunião - buscar_proposta → geração de proposta PDF
<protocolo_de_decisao>
fixo
Sempre igual em todos os clientes. Define a lógica de execução e o retorno ao WR. Ações independentes podem ser paralelas — ações dependentes executam em sequência.
1) Leia ORDEM DO ATENDENTE e MEMÓRIA TÉCNICA. 2) Identifique o tipo de relação entre as ações: → independentes (resultado de uma não afeta a outra) ex: salvar CRM + buscar DB → executa em paralelo → dependentes (uma precisa do resultado da outra) ex: buscar catálogo → registrar interesse → sequência 3) Se falta execução → retorne ferramenta + parâmetros. 4) Se já tem resultado suficiente → retorne agent_wr.
<restricoes>
fixo
Sempre ao final. Schema JSON vai no response_schema do body — mais estrito que o do WR para evitar parametros vazios.
NUNCA invente dados. NUNCA altere retorno de ferramenta. NUNCA responda diretamente ao lead. Budget total estimado do OP: identidade + missao ~60 tokens ferramentas ~120 tokens ← varia por cliente protocolo_decisao ~100 tokens restricoes ~40 tokens ──────────────────────────────── total ~320 tokens
04 · pipeline de análise do histórico
Dois estágios bem separados. LLM 1 extrai — LLM 2 constrói. A qualidade do prompt final depende 100% do schema que a LLM 1 é instruída a extrair.
DB
Exportar e filtrar histórico
Query no messages agrupando por contact_id. Deletar conversas com menos de 5 interações — são ruído, não padrão.
SELECT contact_id, role, content, created_at FROM messages WHERE client_id = '[id]' GROUP BY contact_id HAVING COUNT(*) >= 5 ORDER BY contact_id, created_at
÷
Dividir em batches de 20 conversas
Cada batch contém 20 conversas completas. Esse tamanho equilibra qualidade de extração e custo de tokens por requisição.
Batches menores → mais precisão, mais chamadas de API
Batches maiores → mais ruído por batch, menos controle
20 conversas é o ponto de equilíbrio validado
L1
LLM 1 · extrai schema por batch
Recebe 20 conversas, ignora spam e interações sem engajamento real, extrai os 4 blocos do schema. Repete até processar todos os batches. Salva cada resultado no banco de resumos.
{ "atendimento": { "perguntas_mais_frequentes": [], "como_o_vendedor_respondeu": [], "tom_usado": "" }, "qualificacao": { "como_bant_surgiu_naturalmente": [], "momento_da_conversa": "", "o_que_destravou_o_lead": [] }, "conversao": { "sinal_de_interesse_real": [], "o_que_o_vendedor_fez_quando_percebeu": [], "como_passou_pro_vendedor": "" }, "linguagem": { "vocabulario_do_lead": [], "vocabulario_do_vendedor": [], "objecoes_e_contornos": [] } }
mapeamento schema → seções do WR
atendimento → fluxo_conversa_natural
qualificacao → bant
conversao → quando_acionar_operacional

L2

Lê o banco de resumos acumulados pela LLM 1, consolida padrões repetidos entre batches e preenche as seções do WR marcadas como "do histórico". Entrega o rascunho estruturado para validação no briefing.
Seções preenchidas automaticamente: estilo_resposta, fluxo_conversa, bant, objecoes, quando_acionar_op (parcial)
Seções que ficam para o briefing: identidade, missao, ferramentas do OP, limites comerciais
✓
Rascunho do WR pronto para o briefing
O briefing com o cliente vira uma sessão de 30 min de validação — não uma entrevista de suposições. As seções que vêm do histórico chegam pré-preenchidas com dados reais do negócio dele.
